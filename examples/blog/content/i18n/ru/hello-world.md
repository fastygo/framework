# Привет, мир

Этот пост — короткая экскурсия в то, как построен демонстрационный блог на `fastygo/framework`.

В статье не используется CMS, не нужны внешние API, а исходные тексты лежат прямо в репозитории рядом с кодом. Изменения в markdown автоматически отражаются в рендере, потому что контент собирается вместе с приложением и отдаётся в том же бинарнике.

Если говорить проще: вы пишете текст в обычный `.md`, добавляете запись в реестр постов — и получаете готовую SSR-страницу.

## Как устроен рендер блога

1. На старте приложения список постов читается из встроенных markdown-файлов по локалям.
2. `pkg/content-markdown` конвертирует их в HTML и сохраняет в памяти.
3. На каждый запрос шаблон просто берёт уже готовый HTML из кэша и отдаёт его.

Такой подход даёт две практические выгоды:

- меньше нагрузки на каждый запрос,
- более стабильное поведение под нагрузкой.

### Рендер на старте

Конвертация выполняется один раз при запуске приложения. Это убирает накладные расходы в runtime и делает время ответа предсказуемее.

### Кэш в памяти

Кэшированный HTML особенно полезен для часто открываемых статей. Повторные чтения уже не запускают новый markdown-парсинг.

## Почему это удобно для команды

- Контент и код версиируются вместе.
- Логику рендера можно быстро отладить: markdown → HTML cache → templ.
- Не нужен отдельный сервис для хранения статей.

## Что дальше

Добавьте новый файл в `content/i18n/<locale>/<slug>.md`, зарегистрируйте slug в `internal/site/blog/feature.go`, обновите шаблоны при необходимости и перезапустите сервер.
